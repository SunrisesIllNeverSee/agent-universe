## Issues Found

### Priority 1 — Functional Bugs

**Commission payouts never hit treasury**
`record_commission()` calculates `commission_amount`, writes `status: "pending"` to kassa.db, creates a seed — but never calls `economy.treasury.credit()`. The referrer's balance never increases. Recruitment (same mechanic) correctly calls `treasury.credit()` on line 3594. Commission is missing the same call.

**Product review rewards never hit treasury**
`approve_product_review()` logs `product_review_reward` to audit when approved but never calls `economy.treasury.credit()`. The reward field exists in the review record but nothing pays out. Approval is a paper transaction only.

### Priority 1 — Security

**Operator endpoints have no authentication**
`/api/operator/stats`, `/api/operator/audit`, `/api/operator/contacts`, `/api/mpp/credit`, `/api/operator/reviews` — zero auth. Anyone who knows the URL can read all contact submissions, view full audit history, or credit any agent's balance. Railway domain obscurity is the only protection.

### Priority 2 — Correctness

**Chain withdrawals debit treasury but deliver nothing**
`/api/economy/withdraw` correctly debits the agent treasury (permanent balance reduction) and calls `chain_router.transfer()` — but all three chain adapters return `"note": "Production: submit to ... RPC. This is a governed transaction stub."` The money leaves the ledger, the transfer never executes. Should block treasury debit until chain transfer is confirmed or switch to a pending/escrow model.

**Mission `mission_id` field is always None**
Live missions have a proper `id` field (`mission-ea37db39` etc.) but `mission_id` is `None` in all 3 records. Server code uses both fields interchangeably in different endpoints. Lookups using `mission_id` silently fail.

### Priority 3 — Architectural Gaps

**Seeds are undertriggered**
14 seeds vs 22,271 audit events. `create_seed()` is called from KA§§A posts, registration, governance docs, and a few other places — but not from slots, missions, most messages, or metrics. The provenance system works correctly but covers ~15% of system activity instead of the claimed "every interaction."

**In-memory MPP challenge store**
`_mpp_challenges` is a module-level dict. Every Railway deploy (triggered by `git push`) wipes all pending challenges. Agents mid-payment-flow during a deploy get `"Challenge not found or expired"` on retry. Acceptable for now but challenges should move to SQLite or Redis.

**JSONL/SQLite split on KA§§A posts**
`kassa_posts.jsonl` (10 legacy posts) and `kassa.db` (current SQLite store). The API reads from SQLite only. The 10 JSONL posts are orphaned — invisible through the API unless migrated. Old content exists but the live system cannot serve it.

**Metrics.json is empty in production**
`metrics.json` has no agent entries because nothing auto-populates it. `/api/metrics/agent` must be called explicitly per agent to record data. Flame review, leaderboard, tier calculation all work correctly — there's just no live data feeding them.

**GovernanceProxy is fragile**
`runtime.check_action()` creates an anonymous class with only 3 attributes (mode, posture, role) and passes it to `check_action_permitted()`. This works because governance.py only accesses those 3 today. If any future governance check accesses `governance.domain` or another `GovernanceStateData` field, it will raise `AttributeError`.

**Dual Stripe webhook endpoints**
`/api/connect/webhooks` handles V2 thin events. `/api/kassa/webhooks/stripe` handles V1 checkout events. Different purposes, both need separate Stripe registrations. Naming is confusing and overlap risk exists if Stripe config drifts.

## Structural Observations

1. **server.py needs splitting.** At 4,582 lines it's unnavigable. Natural seams: `kassa_routes.py`, `economy_routes.py`, `governance_routes.py`, `provision_routes.py`, `connect_routes.py`. Each already imports its own store/module.

2. **The audit trail is the strongest part of the system.** 22k events, hash-chained, governance-stamped. This is more complete than most production systems.

3. **The economy math is correct and well-tested.** Tier thresholds, fee splits, trial mechanics — all calculated properly. The gap is activation, not design.

4. **Governance is real but checked inconsistently.** `check_action_permitted()` is called from chains but not from economy payouts, slot fills, or KA§§A transactions. The governance check should be upstream of every state-changing operation, not only chain transfers.

5. **fcntl in seeds.py is POSIX-only.** Fine on Railway (Linux). Would break if ever run on Windows.
